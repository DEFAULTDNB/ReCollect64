1st SSB Recomp Released

Smashing Stuff

With the long-awaited Super Smash Bros. decompilation still hovering at 99.7%, today JRickey has used AI to create a working proof of concept named "Battleship" and has released the first public beta versions for PC/Mac/Linux gamers to get their hands on.

Please note that this is not an official Harbour Masters 64 port, so please read the section below and the project's README to grasp what the author is going for here. The quality of the code is unknown, but JRickey has at least been 100% transparent with why and how this was created.

 

Highlights

Classic Mode (1P) stability — the round of crashes and freezes reported on Break the Targets, Board the Platforms, the boss stage wallpaper, the bonus character-select HiScore display, and the stage-clear screen are all fixed. See docs/audit_per_character_bonus_stages_2026-04-29.md for the full sweep across all 12 fighters.

Rendering — Sector Z (Race-to-the-Finish) bumpers no longer render with wrong colors (LP64 stride + MPGroundData byteswap fix). Boss-stage wallpaper materials render correctly.

Windows / D3D11 — G_TX_MIRROR | G_TX_CLAMP wrap mode now implemented (libultraship bump). No regressions to OpenGL / Metal backends.

Project meta — MIT License added; README clarifies the project is unaffiliated with Nintendo and unaffiliated with Harbour Masters (the libultraship / Torch upstream team).

Install

macOS (Apple Silicon and Intel): download BattleShip.dmg, open, drag BattleShip.app into Applications. First launch will prompt for a ROM.

Linux (x86_64, glibc ≥ 2.39, i.e. Debian 13 / Ubuntu 24.04+): download BattleShip-x86_64.AppImage, chmod +x, run.

Windows (10/11): download BattleShip-windows.zip, unzip, run BattleShip.exe.

ROM requirement

This is a clean-room port: no Nintendo content is in the build. On first launch the app will ask you to point it at your own legal NTSC-U v1.0 ROM (SHA-1 e2929e10fccc0aa84e5776227e798abc07cedabf). If you don't own one, you cannot run the game.

Known issues

Latent finds from the per-character audit are catalogued in docs/audit_per_character_bonus_stages_2026-04-29.md. None have a confirmed user-tied reproduction; if you hit a freeze, please file an issue with ssb64.log from the run.

Artificial Intelligence

Controversially, JRickey has made no bones about his use of AI to get this to a playable state. Using Opus 4.6, Opus 4.7, and GPT 5.5 over the course of 25 days, the project has been far from plain sailing, and the author has encountered many issues along the way and spent a lot of his own money on subscriptions and fees to use these LLMs.

This port took just 2 days to get the initial basic "scaffolding" working and a further 20+ days to iron out everything else around it to get it to function as you would expect. As with most AI projects, they're difficult to debug and control if you don't know the ins and outs of the source code and the toolsets required to craft these ports. I'm sure other recomp/port developers will have more to say on this in due time if or when they cast an eye over the code used.

With AI becoming more prevalent, it's certain we will see this happen more and more, but the competency of the authors is where these recomps will stand the test of time or fall at the first hurdle producing updates, mods and improvements. Jnmartin, who has made some incredible N64-to-Dreamcast ports, recently took to BlueSky to voice his use of AI in his recent PC port of Sonic R; however, regardless of the results shown there was a lot of discourse about it.

Understandably, some people hate the use of AI altogether, but I think this appears to demonstrate that these tools could at least prove useful at least for automating the grunt work and countless thankless tasks involved with projects like these. As long as the creators of the source materials (decomps/recomps and tools) are correctly credited, I think it's an interesting little experiment which looks to have worked quite well from an outsiders perspective.
