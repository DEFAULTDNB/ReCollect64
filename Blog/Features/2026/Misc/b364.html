UPDATED SM64 Running on PSX & 3DO

The 3DO One is Not a Port! ~ Jul 15, 2026
SM64 Running on PSX & 3DO

PS & Panasonic Polygons

Last week, Eyepatch Entertainment released a video showcasing Super Mario 64 running on Sony's OG PlayStation console, and today they have also shown footage of various assets pulled from the SM64 ROM running in some capacity on a Panasonic 3DO.

Based on the SM64 decompilation, the PlayStation port is based on the WIP PSX port by Malucard but targeting PSX hardware rather than PSX emulation. The 3DO project is somewhat different in that the 64-bit decompilation's assets are running directly within 3DO's CEL Engine. The authors are currently referring to the 3DO version as a piece of homebrew, rather than a port.

Here is the current progress report for the PS version:

 

Recorded off a CRT to show its really all on hardware and not an emulator. this was my second PS1 adventure after porting my other game/app: HERE.

**Foundation**: Started as a fork of the n64decomp/sm64 decompilation by https://github.com/malucard/sm64-psx by malucart, retargeted to boot on real PS1 hardware instead of N64/emulator.

**Phase 1 — 2MB bring-up (June, builds ~1–1900)**: Spent this whole stretch just getting the game to boot and stay up within the PS1's real 2MB RAM budget. Fixed stale CD-image LBA baked in at compile time, misaligned ISO9660 reads, and wrong CD controller mode causing garbage texture loads. Early debugging was "guess and check against screenshots" for ~68 builds before switching to GDB/backtrace debugging via PCSX-Redux, which is when progress actually started sticking. Most of this era was repeated `MAIN_POOL_LOW_ALLOC`/RAM-overflow assert fights on Bob-omb Battlefield — the literal fight to fit in 2MB. Around build ~1890 the team confirmed explicitly: upstream was built for real 2MB PS1 use, CD loading was the historical blocker.

**Phase 2 — Visual/gameplay polish, (July 1–3)**: Terrain LOD tuning, Peach cutscene framing, first audio investigation (found audio builds overflowed RAM by ~18.5KB). July 2 was the big one: **first successful boot and run on real PS1 hardware**, after fixing an inverted framebuffer flip bug that only showed up on accurate hardware/emulators, not lenient ones. Also fixed Mario's see-through head bug. July 3: root-caused a recurring Bob-omb Battlefield crash (object-pool/coin exhaustion) and added spawn-burst crash guards.

**Phase 3 — (July 4–5)**: Assembled a "Demo1" build for a planned YouTube capture, which kicked off rapid same-day triage of five bugs (castle brightness, missing Chain Chomp, Mario model, star select, texture corruption). This is the pivot point where audio/music became the top priority instead of a "someday" item. July 5: root-caused a permanent castle door-loop/game-over bug as silently dropped collision surfaces (undersized static collision pool, not a warp bug). Same day, hardware audio filter attempt caused erratic behavior on console.

*Phase 4 — Music on real hardware (July 5–6, today)**: July 5, got the first-ever audible hardware music (a Peach-letter fragment) by correctly mapping XA stream files, but playback died on later CD reads. July 6: **root cause nailed down* — every music track built to date was encoded as 8-bit ADPCM, which emulators decode fine but real PS1 hardware only supports 4-bit XA-ADPCM, so music worked in DuckStation/Redux and played as garbage on console. Fix: re-encode all music 4-bit, adjust sector interleave, and add an automated hardware-decodability check so this class of bug can't sneak back in. This is the fix that has music working on console now.


Here is the current progress report for the 3DO version:

 

Super Mario 64's Bob-omb Battlefield and Peach's Castle running on the Panasonic 3DO, a console that never had anything like it. Built from the real Super Mario 64 decompilation assets, rendered entirely on the 3DO's cel engine. No sound yet. This is a hobby port, still in progress.

This video shows it running on real 3DO hardware (Panasonic FZ-10 via a USB optical drive emulator) alongside the Opera emulator, so you can see both.

What's in so far: the full castle exterior and interior, the courtyard, and a walkable Bob-omb Battlefield with enemies, the Chain Chomp, the Big Bob-omb fight, stars, the seesaw bridge, checkerboard elevators and bowling balls — all placed from the game's original level data.

The hard part isn't the geometry, it's the 3DO's fill rate: its cel engine pays per on-screen pixel, so big close-up surfaces are murder. Getting it to hold a framerate on real hardware took a lot of very specific tricks and a lot of builds.

No copyrighted ROMs or assets are distributed — this is a non-commercial technical demo.

Harnessing Homebrew

It should be noted that the 3DO (1994) cannot produce polygons with the same efficiency that the PSX (1994) or N64 (1996) could, so the differences with this 3DO effigy mean that a 3D Mario model is impossible, so a 2D sprite is currently employed with tank controls and buttons mapped for specific moves like jump and ground pound, as well as strafing left and right.

The 3DO version is also not a fork from any other version; it's a brand new implementation that runs directly on 3DO hardware or emulators. The reason a port is not achieveable right now is because the CEL engine graphics co-processor lacks a traditional geometric 3D pipeline, whereas the PlayStation had a dedicated math co-processor called the Geometry Transformation Engine (GTE), and the N64 had the Reality Coprocessor (RCP) to handle complex geometry and advanced lighting.

The PlayStation version is far more akin to ports that we would recognise. It runs with the cut scenes and sound, albeit extremely buggy and unpolished. The developers note VRAM corruption and various bugs in the video that are frequently causing stutters and freezes, but the progress since the initial implementation for emulators in November of last year to getting it running on real hardware just 8 months later is an incredible technical achievement.

It's impressive to see the spoils of the SM64 decompilation project still producing quality content, years after it was completed. Super Mario 64 is becoming the new Doom in that it's almost playable in one form or another on almost every platform imaginable.

UPDATE: 22/07/2026 - The 3DO version has been updated and is looking significantly better with a dev menu, Mario sprite improvements, remade enemies, and sound effects!
